# 🤝 مشارکت در لینگو

از اینکه به مشارکت در لینگو علاقه دارید، از شما سپاسگزاریم! لطفاً برای اطمینان از روندی روان برای مشارکت، این دستورالعمل‌ها را دنبال کنید.

## 📋 قبل از شروع

### ابتدا Issues موجود را بررسی کنید
**بدون بررسی issues باز، مستقیماً اصطلاحات یا سوالات چالشی اضافه نکنید.**

1. به بخش [Issues](https://github.com/homayounmmdy/Lingoo/issues) بروید.
2. به دنبال issues با برچسب‌های زیر بگردید:
   - `good-first-issue`
   - `help-wanted`
   - `community`
   - `first-timers`
3. **در issue کامنت بگذارید** و بگویید که می‌خواهید روی آن کار کنید.
4. منتظر بمانید تا issue به شما واگذار شود یا تأییدیه بگیرید.

### چرا این موضوع مهم است؟
- از کارهای تکراری جلوگیری می‌کند.
- کنترل کیفیت را تضمین می‌کند.
- ساختار JSON را یکپارچه نگه می‌دارد.
- به نگهدارندگان پروژه اجازه می‌دهد مشارکت‌ها را پیگیری کنند.

## 🔧 روند مشارکت

### مرحله 1: فورک و کلون
```bash
# ابتدا مخزن را روی گیت‌هاب فورک کنید، سپس:
git clone https://github.com/YOUR-USERNAME/Lingoo
cd Lingoo
```

### مرحله 2: یک شاخه جداگانه ایجاد کنید
**همیشه برای مشارکت خود یک شاخه جدید ایجاد کنید**

```bash
# برای اصطلاحات
git checkout -b add-idiom-[idiom-name]

# برای سوالات چالشی
git checkout -b add-trivia-[question-topic]

# برای رفع باگ‌ها
git checkout -b fix-[bug-description]

# برای ویژگی‌های جدید
git checkout -b feature-[feature-name]
```

### مرحله 3: Issue را به دقت بخوانید
- دقیقاً متوجه شوید که چه چیزی نیاز است.
- هر گونه نیاز خاص ذکر شده را بررسی کنید.
- ساختار JSON را دقیقاً همان طور که مشخص شده دنبال کنید.
- در صورت لزوم، ترجمه فارسی را تأیید کنید.

### مرحله 4: تغییرات خود را اعمال کنید
```bash
# تغییرات خود را اضافه کنید
git add .

# با یک پیام واضح Commit کنید
git commit -m "add: [توضیحاتی در مورد آنچه اضافه کردید]"

# به فورک خود Push کنید
git push origin [your-branch-name]
```

### مرحله 5: یک Pull Request باز کنید
1. به مخزن اصلی بروید.
2. روی "New Pull Request" کلیک کنید.
3. شاخه خود را انتخاب کنید.
4. **issue را لینک کنید** با نوشتن `Closes #issue-number` در توضیحات PR.
5. برای بررسی ارسال کنید.

## 📝 چه چیزی را مشارکت دهید

### 🐛 گزارش باگ
- بررسی کنید که باگ قبلاً گزارش نشده باشد.
- از الگوی گزارش باگ استفاده کنید.
- مراحل بازتولید باگ را شامل کنید.
- در صورت مفید بودن، اسکرین‌شات اضافه کنید.

### 💡 پیشنهاد ویژگی
- درخواست‌های ویژگی موجود را بررسی کنید.
- ویژگی را به وضوح توضیح دهید.
- مزایا را برای کاربران فارسی‌زبان توضیح دهید.
- در مورد ایده‌های پیاده‌سازی دقیق باشید.

### 🎮 سوالات چالشی
**فقط پس از واگذاری issue به شما:**
- به فایل `/public/community/english-trivia.json` اضافه کنید.
- قالب دقیق JSON را دنبال کنید.
- صحیح بودن همه پاسخ‌ها را تأیید کنید.
- یک واقعیت جالب آموزشی را شامل کنید.

### 📚 اصطلاحات
**فقط پس از واگذاری issue به شما:**
- به فایل `/public/community/idioms.json` اضافه کنید.
- قالب دقیق JSON را دنبال کنید.
- از ترجمه دقیق فارسی اطمینان حاصل کنید.
- یک جمله مثال طبیعی ارائه دهید.

## ❌ چه کارهایی را نباید انجام داد

- **بدون issue** اصطلاح اضافه نکنید.
- روی شاخه اصلی (main) کار نکنید.
- تغییرات نامرتبط را در یک PR ارسال نکنید.
- نیازهای issue را نادیده نگیرید.
- بدون تست محلی، ارسال نکنید.

## ✅ چک‌لیست Pull Request

قبل از ارسال، تأیید کنید:

- [ ] issue را با دقت خوانده‌ام.
- [ ] یک شاخه جداگانه ایجاد کرده‌ام.
- [ ] تغییرات من با نیازهای issue مطابقت دارد.
- [ ] ساختار JSON معتبر است.
- [ ] پیام commit من واضح است.
- [ ] issue را در توضیحات PR لینک کرده‌ام.

## 🚀 راه‌اندازی محیط توسعه

```bash
# نصب وابستگی‌ها
npm install

# اجرای محلی
npm run dev

# بررسی خطاها
npm run lint

# ساخت برای تست محیط تولید
npm run build
```

## 💬 نیاز به کمک دارید؟

- **دوباره issue را بخوانید** - معمولاً نیازها آنجا نوشته شده است.
- **در issue کامنت بگذارید** - برای شفاف‌سازی سوال بپرسید.
- **README.md را بررسی کنید** - برای آشنایی با ساختار پروژه.
- **یک بحث (discussion) باز کنید** - برای سوالات عمومی.

## 📌 به خاطر داشته باشید

1. **همیشه ابتدا issues را بررسی کنید** - حدس نزنید که چه چیزی نیاز است.
2. **شاخه‌های جداگانه ایجاد کنید** - هر شاخه فقط برای یک مشارکت.
3. **نیازها را با دقت بخوانید** - دقیقاً همان چیزی که خواسته شده را دنبال کنید.
4. **PR خود را به issue مرتبط کنید** - تا نگهدارندگان بدانند چه چیزی را حل می‌کند.
5. **صبور باشید** - بررسی‌ها زمان می‌برند.

## ⭐ مشارکت‌کنندگان بار اول؟

به دنبال issues با برچسب‌های زیر بگردید:
- `good-first-issue`
- `beginner-friendly`
- `first-timers`

این issues دستورالعمل‌های واضحی دارند و برای شروع عالی هستند.
